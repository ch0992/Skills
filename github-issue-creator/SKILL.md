# GitHub Issue Creator

프로젝트 CONTRIBUTING.md 규칙에 따라 이슈를 분류하고, 제목·본문을 생성한 뒤 GitHub에 등록한다.

## 이슈 유형 분류

사용자의 요청을 다음 세 가지 중 하나로 판단한다.

### 1. 부모 이슈 — Phase 단위 묶음 작업
큰 기능 단위, Phase 수준의 작업 묶음.

**제목:**
```
[v{버전} Phase{N}] {작업 내용}
```
**라벨:** `enhancement`
**마일스톤:** 제목의 Phase 번호와 반드시 일치

### 2. 서브 이슈 — 단위 작업
부모 이슈 하위의 구체적 단일 작업.

**제목:**
```
[Sub #{부모이슈번호} | {작업유형}] {작업 내용}
```
작업유형: `bugfix` | `enhancement` | `refactor` | `chore`

### 3. 버그 리포트
재현 가능한 버그.

**제목:**
```
[Bug] {버그 내용}
```
**라벨:** `bug`

---

## 분류 판단 기준

| 상황 | 유형 |
|---|---|
| 부모 이슈 번호를 언급 | 서브 이슈 |
| "버그", "crash", "안 됨", "오류", "에러" | 버그 리포트 |
| Phase 단위 큰 작업 묶음 | 부모 이슈 |
| 그 외 단독 작업/개선 | 서브 이슈 (부모 없으면 부모 이슈로) |

## 본문 템플릿

### 부모 이슈 / 서브 이슈
```markdown
## 개요
{이 작업이 필요한 이유를 1~2줄로}

## 작업 내용
- [ ] 세부 작업 1
- [ ] 세부 작업 2

## 완료 기준
{이슈가 close되기 위한 구체적 조건}

## 산출물
{생성되는 파일, 화면, 문서 등}

## 선행 조건
- {없으면 "없음"}
```

### 서브 이슈 (bugfix 포함)
```markdown
## 개요
{무엇을 수정/추가하는지}

## 원인 (bugfix인 경우)
{버그 원인 분석}

## 작업 내용
- [ ] 세부 작업 1
- [ ] 세부 작업 2

## 수정 파일
- `lib/...`

## 완료 기준
{close 조건}

## 선행 조건
- #{부모이슈번호} 참고
```

### 버그 리포트
```markdown
## 버그 설명
{무엇이 잘못 동작하는지}

## 재현 방법
1.
2.

## 예상 동작
{어떻게 동작해야 하는지}

## 실제 동작
{실제 동작}

## 환경
- 기기:
- OS 버전:
- 앱 버전:

## 스크린샷
{있으면 첨부}
```

## 실행 절차

1. **정보 파악** — 사용자 설명에서 이슈 유형, 버전/Phase, 부모 번호, 작업 내용을 추출한다.
   - 불명확한 정보가 있을 때만 질문 1~2개로 확인한다. 과도하게 묻지 않는다.

2. **마일스톤 조회** — 부모 이슈라면 GitHub MCP로 열린 마일스톤 목록을 확인한다.
   - Phase 번호와 일치하는 마일스톤을 선택한다.

3. **이슈 초안 제시** — 제목 / 라벨 / 마일스톤 / 본문을 사용자에게 보여준다.

4. **승인 후 생성** — 사용자가 확인하면 GitHub MCP로 이슈를 생성한다.
   - 항상 `owner`, `repo` 를 명시적으로 전달한다.

5. **완료 보고** — 생성된 이슈 URL과 번호를 출력한다.

6. **서브 이슈 연결 안내** — 서브 이슈라면 부모 이슈의 Sub-issues 패널에 연결하도록 안내한다.

---

## 작성 원칙

- 이슈 제목은 간결하고 동사형으로 마무리 (예: `~추가`, `~개선`, `~수정`)
- close 시 반드시 코멘트에 산출물 링크(커밋, 파일 등) 첨부 필요 — 사용자에게 안내
- GitHub MCP 호출 시 `owner` / `repo` 항상 명시
