# GitHub Issue Handler

GitHub 이슈의 전체 생명주기를 관리한다.

| 단계 | 액션 | 트리거 예시 |
|------|------|------------|
| 생성 | 이슈 분류 → 초안 → 등록 | "이슈 만들어줘", "이슈 써줘" |
| 업데이트 | 진행 상황 코멘트 추가 | "이슈 코멘트 달아줘", "진행 상황 업데이트" |
| 검수 | 완료 코멘트 작성 | "anti 완료했어", "#XX 완료 코멘트" |
| 클로즈 | 코멘트 확인 후 클로즈 | "이슈 닫아줘", "클로즈해줘" |

---

## Action 1. 이슈 생성

### 이슈 유형 분류

| 유형 | 제목 형식 | 라벨 |
|------|----------|------|
| 부모 이슈 | `[v{버전} Phase{N}] {작업 내용}` | `enhancement` |
| 서브 이슈 | `[Sub #{부모번호} | {bugfix|enhancement|refactor|chore}] {작업 내용}` | — |
| 버그 리포트 | `[Bug] {버그 내용}` | `bug` |

### 분류 기준

| 상황 | 유형 |
|------|------|
| 부모 이슈 번호 언급 | 서브 이슈 |
| "버그", "crash", "안 됨", "오류", "에러" | 버그 리포트 |
| Phase 단위 큰 작업 묶음 | 부모 이슈 |
| 그 외 | 서브 이슈 (부모 없으면 부모 이슈) |

### 본문 템플릿

**부모 이슈 / 서브 이슈:**
```markdown
## 개요
{이 작업이 필요한 이유 1~2줄}

## 작업 내용
- [ ] 세부 작업 1
- [ ] 세부 작업 2

## 완료 기준
{이슈가 close되기 위한 구체적 조건}

## 산출물
{생성되는 파일, 화면, 문서 등}

## 선행 조건
- {없으면 "없음"}
```

**버그 리포트:**
```markdown
## 버그 설명
{무엇이 잘못 동작하는지}

## 재현 방법
1.
2.

## 예상 동작 / 실제 동작

## 환경
- 기기 / OS 버전 / 앱 버전

## 스크린샷
```

### 실행 절차

1. 사용자 설명에서 유형·버전·Phase·부모 번호·작업 내용 추출 (불명확한 것만 질문, 최대 2개)
2. 부모 이슈라면 마일스톤 조회 → Phase 번호 일치하는 마일스톤 선택
3. 제목 / 라벨 / 마일스톤 / 본문 초안 제시 → 사용자 승인
4. `issue_write(owner, repo, ...)` 로 등록
5. 생성된 이슈 URL + 번호 출력
6. 서브 이슈라면 부모 이슈 Sub-issues 패널 연결 안내

---

## Action 2. 이슈 코멘트 업데이트

진행 중인 이슈에 중간 경과 또는 부분 완료 내용을 코멘트로 추가한다.

### 코멘트 형식

```markdown
## 진행 상황 업데이트

**작업 내용:**
- [x] 완료된 항목
- [ ] 남은 항목

**변경 파일:**
- `경로/파일명`

**비고:**
{특이사항 또는 다음 단계}
```

### 실행 절차

1. 이슈 번호 확인 (없으면 요청)
2. `issue_read(owner, repo, issue_number)` 로 현재 상태 확인
3. 코멘트 초안 제시 → 사용자 승인
4. `add_issue_comment(owner, repo, issue_number, body)` 로 추가

---

## Action 3. 완료 검수 코멘트

작업 완료 후 이슈를 닫기 전에 완료 내용을 코멘트로 기록한다.
**클로즈는 이 코멘트 작성 후 별도로 사용자 확인을 받아야 한다.**

### 완료 코멘트 형식

```markdown
## ✅ 작업 완료

**완료 항목:**
- [x] 세부 작업 1
- [x] 세부 작업 2

**산출물:**
- 커밋: {커밋 해시 또는 링크}
- 변경 파일: `경로/파일명`

**완료 기준 충족 여부:**
- [x] {완료 기준 1}
- [x] {완료 기준 2}

**비고:**
{특이사항, 후속 작업 필요 여부 등}
```

### 실행 절차

1. 이슈 번호 확인
2. `issue_read(owner, repo, issue_number)` 로 원래 완료 기준 확인
3. `tasks/reports/` 에서 해당 작업 리포트 확인 (있으면 내용 참고)
4. 완료 코멘트 초안 제시 → 사용자 승인
5. `add_issue_comment(...)` 로 코멘트 추가
6. 클로즈 여부를 **별도로** 사용자에게 확인

---

## Action 4. 이슈 클로즈

완료 코멘트가 달려 있는지 반드시 확인 후 클로즈한다.

### 클로즈 전 검수 체크리스트

```
[ ] 완료 코멘트가 이슈에 달려 있는가?
[ ] 완료 기준이 모두 체크되었는가?
[ ] 관련 커밋/파일 링크가 코멘트에 포함되어 있는가?
```

### 실행 절차

1. `issue_read(owner, repo, issue_number)` 로 코멘트 목록 확인
2. 위 체크리스트를 사용자에게 보여주고 확인
3. 미충족 항목이 있으면 → Action 3 먼저 진행 안내
4. 모두 충족 시 → `issue_write(owner, repo, issue_number, state="closed")` 실행
5. 클로즈 완료 메시지 출력

---

## 공통 원칙

- 이슈 제목은 간결하고 동사형으로 마무리 (예: `~추가`, `~개선`, `~수정`)
- GitHub MCP 호출 시 `owner` / `repo` 항상 명시
- **클로즈는 항상 사용자 최종 확인 후 진행 — 자동 클로즈 금지**
- 코멘트 없이 클로즈 요청이 오면 Action 3 먼저 진행 안내
